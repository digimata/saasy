# T-0014: Feature entitlement matrix

Define a typed entitlement system that maps plan tiers (hobby, pro, ultra) to feature access and limits.

## Scope

- Entitlement catalog: per-plan feature flags and numeric limits (e.g. max projects, max members)
- A `can(workspaceId, feature)` or `limit(workspaceId, feature)` style API
- Server-side enforcement at API boundaries
- Client-side access for UI gating (greyed-out features, upgrade prompts)
- Should compose with the versioned plan catalog in `@repo/billing`

## Open questions

- Hard enforcement vs soft limits (block vs warn)?
- Where do entitlements live — static in code or DB-backed for runtime changes?
- How does this interact with plan version? (e.g. v1 Pro gets 10 projects, v2 Pro gets 20)
