## Objective

All authenticated routes should be prefixed with the active workspace slug: `/{workspace_slug}/settings`, `/{workspace_slug}/members`, etc. Standard multi-tenant URL scheme.

## Current state

Dashboard routes are at `/`, `/settings`, `/members`, `/billing` with no workspace context in the URL. The active workspace is tracked server-side via `session.activeWorkspaceId`.

## Target state

- `/{slug}/` — workspace dashboard (overview)
- `/{slug}/settings` — workspace settings
- `/{slug}/members` — workspace members
- `/{slug}/billing` — workspace billing
- Switching workspaces changes the URL prefix
- Navigating to `/{slug}/...` where slug doesn't match active workspace → switch active workspace (if user is a member) or 404
- `/` (bare root) → redirect to `/{active_workspace_slug}/`

## Route structure

```
app/
  (auth)/          ← unchanged (sign-in, sign-up, accept-invitation)
  onboard/         ← unchanged (no workspace yet)
  [workspace]/     ← dynamic segment replaces current (dash)
    layout.tsx     ← validate slug, set active workspace, render sidebar
    page.tsx       ← overview
    settings/
    members/
    billing/
```

## Key considerations

- Middleware needs to resolve workspace slug from the URL, not just check session cookie
- Sidebar nav links must include `/{slug}/` prefix
- Onboard page redirect should go to `/{new_slug}/` after workspace creation
- Sign-in redirect should resolve to `/{active_slug}/` not just `/`
- Workspace switcher should navigate to `/{new_slug}/` on switch
- Need to handle: slug not found, user not a member of that workspace, no active workspace
