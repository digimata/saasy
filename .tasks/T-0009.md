## Objective

All authenticated routes should be prefixed with the active workspace slug: `/{workspace_slug}/settings`, `/{workspace_slug}/members`, etc. Standard multi-tenant URL scheme.

## Current state

Dashboard routes are at `/`, `/settings`, `/members`, `/billing` with no workspace context in the URL. The active workspace is tracked server-side via `session.activeOrganizationId`.

## Target state

- `/{slug}/` — workspace dashboard (overview)
- `/{slug}/settings` — workspace settings
- `/{slug}/members` — workspace members
- `/{slug}/billing` — workspace billing
- Switching workspaces changes the URL prefix
- Navigating to `/{slug}/...` where slug doesn't match active workspace → switch active workspace (if user is a member) or 404
- `/` (bare root) → redirect to `/{active_workspace_slug}/`

## Implementation Plan

### Phase 1: Root redirect page

Create `app/page.tsx` — server component at the root level (outside route groups).

1. Read session via `auth.api.getSession`
2. No session → redirect `/sign-in`
3. No `activeOrganizationId` → redirect `/onboard`
4. Query workspace slug from `workspaces` table by `activeOrganizationId`
5. Redirect to `/{slug}/`

Middleware stays unchanged — no DB calls in middleware. The root page handles the redirect. This only fires once per navigation to bare `/`.

### Phase 2: Route restructure — `(dash)` → `[workspace]`

Rename `app/(dash)/` → `app/[workspace]/`. All child routes move as-is.

**`app/[workspace]/layout.tsx`** — the critical file:

1. Read `workspace` param from dynamic segment
2. Check session (same as current layout)
3. No session → redirect `/sign-in`
4. Look up workspace by slug in DB (join with `memberships` to verify access)
5. Slug not found OR user not a member → `notFound()` (don't leak existence)
6. If workspace ID differs from `session.activeOrganizationId` → call `auth.api.setActiveOrganization` to sync
7. Render sidebar + header + children, passing slug via context

Reserved paths (`/sign-in`, `/sign-up`, `/onboard`, `/api/*`) are safe — Next.js static routes and route groups take priority over dynamic segments.

### Phase 3: Workspace slug context

New `lib/workspace-context.tsx`:

```tsx
const WorkspaceSlugContext = createContext<string>("");
export const WorkspaceSlugProvider = WorkspaceSlugContext.Provider;
export function useWorkspaceSlug() { return useContext(WorkspaceSlugContext); }
```

Rendered by `[workspace]/layout.tsx` wrapping children.

### Phase 4: Sidebar nav links

`components/sidebar.tsx`:

- Read slug from `useWorkspaceSlug()`
- Prefix all `href`s: `/${slug}`, `/${slug}/projects`, `/${slug}/orders`
- Settings link: `/${slug}/settings`
- Update active detection: strip `/${slug}` prefix before matching, or match against full prefixed paths

### Phase 5: Workspace switcher navigation

`components/workspace-switcher.tsx` + `components/auth/organization/organization-switcher.tsx`:

The switcher already supports `pathMode: "slug"` — when enabled, `switchOrganization` navigates to `/${organization.slug}`. Two options:

**Option A** (minimal): Override `switchOrganization` behavior in `workspace-switcher.tsx` by passing `onSetActive` that navigates to `/${org.slug}/`.

**Option B** (cleaner): Enable `pathMode: "slug"` in the auth provider's organization config. The switcher already checks `pathMode === "slug"` and navigates accordingly.

Going with **Option A** — keep the change localized, avoid re-configuring the global auth provider.

In `workspace-switcher.tsx`:
- After `switchOrganization` succeeds, call `router.push(/${org.slug}/)` and set `activeOrganizationId` in sync.

In `organization-switcher.tsx`:
- Pass `onSetActive` from `workspace-switcher.tsx` that handles the navigation.

### Phase 7: Test updates

### Edge cases

- **Slug not found / not a member**: `notFound()`, don't distinguish
- **Concurrent tabs**: layout syncs `activeOrganizationId` on each load — acceptable for MVP
- **Slug rename**: navigate to new slug after save in general settings
- **No active workspace**: root page redirects to `/onboard`
- **Delete workspace**: redirect to `/` → root page sends to next workspace or `/onboard`

### Sequence

Phases 1-4 are the critical path (can't ship without all four). Phase 5 is required for workspace switching to work properly. Phase 6 is polish. Phase 7 is validation.
